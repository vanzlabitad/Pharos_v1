# Pharos

[![Weekly data refresh](https://github.com/Vanz23-23/Pharos_v1/actions/workflows/refresh.yml/badge.svg)](https://github.com/Vanz23-23/Pharos_v1/actions/workflows/refresh.yml)
[![Secret scan](https://github.com/Vanz23-23/Pharos_v1/actions/workflows/secret-scan.yml/badge.svg)](https://github.com/Vanz23-23/Pharos_v1/actions/workflows/secret-scan.yml)

A pharmacovigilance signal-detection platform that ingests adverse event reports from OpenFDA, computes disproportionality statistics (ROR + PRR with the EVANS criterion), and surfaces drug safety signals in a public dashboard with plain-language AI summaries.

> **Status:** MVP build in progress. Pipeline, analysis, and dashboard scaffolding complete; methodology report and CI rendering pending. See [§9 of CLAUDE.md](CLAUDE.md#9-feature-scope) for the live punch list.

---

## What it does

For every drug-reaction pair in the ingested FAERS sample, Pharos computes a Reporting Odds Ratio (ROR) and a Proportional Reporting Ratio (PRR), then flags pairs that pass the **EVANS criterion** — ROR lower 95% CI > 1, PRR ≥ 2, *n* ≥ 3, χ² ≥ 4 — as safety signals worth a closer look. Every flagged drug gets a 2–3 sentence plain-language summary generated by Gemini at refresh time, so non-scientific readers can understand what a signal does and does not mean.

The dashboard updates weekly from a GitHub Actions cron; data is shipped as static JSON, the Quarto methodology report is rendered to HTML, and Vercel redeploys on push. There is no live API in production — the architecture is deliberately boring.

## Architecture

```
        OpenFDA API
             │
             ▼
   Python ingest (requests, pandas)
             │
             ▼
        SQLite (db/pharos.db)
             │
             ▼
   ROR + PRR (scipy, statsmodels)
             │
             ▼
   Static JSON exports + Gemini summaries
             │
             ▼
   Next.js dashboard (Vercel) + Quarto report
```

Weekly GitHub Actions workflow ([`.github/workflows/refresh.yml`](.github/workflows/refresh.yml)) drives the whole pipeline and commits the output back to the repo. The Gemini API key never leaves CI — the frontend reads only pre-generated static JSON. See [§10 of CLAUDE.md](CLAUDE.md#10-ai-plain-language-summaries) for the key-safety policy.

## Quickstart

Requires Python 3.11+ and Node 20+.

```bash
# 1. Clone and install Python deps
git clone https://github.com/Vanz23-23/Pharos_v1.git
cd Pharos_v1
pip install -r requirements.txt

# 2. (Optional) set API keys for live ingest + summaries
#    Without these, you can still run tests against the committed fixtures.
export OPENFDA_API_KEY=...
export GEMINI_API_KEY=...

# 3. Run the weekly pipeline end-to-end (ingest → ROR/PRR → JSON → summaries)
python -m pipeline.run_refresh

# 4. Run the Quarto methodology report
quarto render analysis/report.qmd

# 5. Boot the dashboard
cd dashboard
npm install
npm run dev          # http://localhost:3000
```

To enable the pre-commit secret scanner locally:

```bash
git config core.hooksPath .githooks
```

## Methodology in one paragraph

Pharos treats a flagged signal as a hypothesis, not a verdict. Disproportionality on FAERS is useful for signal *generation* and tells you nothing about causation. Reports are aggregated under a canonical generic name via a curated alias table ([`pipeline/drug_aliases.json`](pipeline/drug_aliases.json)); the background denominator is "all other drugs" globally, with no ATC or time stratification; zero contingency cells get Yates' 0.5 continuity correction by default; a pair must have *a* ≥ 3 observed co-occurrences before it is even computed (the EVANS floor). Known caveats — notoriety bias, Weber effect, indication confounding, masking, stimulated reporting, no exposure denominator, brand/generic ambiguity — are all live and documented. Full method, formulae, and references are in [`analysis/report.qmd`](analysis/report.qmd) and [§8 of CLAUDE.md](CLAUDE.md#8-analytical-core--rorprr-disproportionality).
